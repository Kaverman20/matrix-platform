почему так произошло → как починили**. Цель — чтобы команда понимала контекст,
а не разбиралась заново. Новые записи добавляем сверху.

## 2026-06-16 — P1.4/P1.5 завершены: SDK только в matrix-core + зачистка заглушек

**Контекст.** Завершена изоляция Matrix SDK: вся рантайм-логика вынесена в
`packages/matrix-core`, `apps/` импортирует из `matrix-js-sdk` только типы.
Граница закреплена ESLint-правилом (`@typescript-eslint/no-restricted-imports`
с `allowTypeImports` для `apps/**`) — любой value-импорт SDK в приложении теперь
ошибка линтера. Core unit-тестов: 71.

**Намеренный мёртвый код (чтобы аудит не считал находкой).**
- `enableEncryption()` в `packages/matrix-core/src/client/startClient.ts` —
  намеренно не вызывается (см. запись про E2EE ниже). Не баг.
- `apps/surf-chat-web/src/features/selection/` и `src/components/` — пустые
  папки-маркеры будущих фич, только README (помечены `(planned)`). Не забытый код.

**Удалено.** Пакет `packages/shared` (`@matrix-platform/shared`) — содержал
единственный неиспользуемый тип `Maybe<T>` и ни на что не был подключён (даже не
в зависимостях web-app). Удалён целиком как мёртвый; если `Maybe<T>` понадобится —
это одна строка.

## 2026-06-15 — E2EE отложен (crypto-wasm уже ленивый, не на критическом пути)

**Контекст (P1.1).** Стоял вопрос судьбы E2EE: crypto-wasm весит ~5.5 MB, ранее
`initRustCrypto` грузил его синхронно на логине и убивал перф.

**Что выяснили (замерено на проде).**
- На старте грузится только `index-*.js`. Чанк `rust-crypto` и `matrix_sdk_crypto_wasm`
  **не загружаются вообще** (`performance.getEntriesByType('resource')` их не содержит) —
  matrix-js-sdk тянет crypto динамически, а `enableEncryption()` (→ `initRustCrypto`)
  **нигде не вызывается**. То есть wasm лежит в `dist`, но в рантайм не попадает →
  штрафа на загрузку/перф сейчас нет.
- Из 6 присоединённых комнат пользователя `@f.foxhound` **0 зашифрованных**
  (проверено через `GET /rooms/{id}/state/m.room.encryption/`).

**Решение.** E2EE **отложен**. Кода не меняем: crypto уже ленивый и не нужен,
пока нет зашифрованных комнат. Когда E2EE понадобится — вызвать
`enableEncryption(client)` (`packages/matrix-core/src/client/startClient.ts`)
лениво при первом входе в зашифрованную комнату (и в воркере), а не на бут-пути.
Остаточный пункт (низкий приоритет): wasm всё ещё попадает в `dist` (размер
деплоя, не скорость) — при желании можно исключить crypto из сборки.

## 2026-06-15 — Разделитель непрочитанного не появлялся (устаревший кэш)

**Проблема.** При входе в комнату с непрочитанными разделитель «Новые
сообщения» не показывался, и чат открывался внизу, хотя бейдж непрочитанных был.

**Причина.** `useFirstUnread` вычислял первое непрочитанное по мемоизированному
массиву `messages`, а он при открытии комнаты мог быть устаревшим (из
предзагрузки — до прихода новых сообщений). В устаревшем массиве read-marker
оказывался последним элементом → «непрочитанных нет» → разделителя нет.

**Решение.** Считаем первое непрочитанное по **живому таймлайну** комнаты
(`room.getLiveTimeline().getEvents()`) — он всегда актуален. Плюс доскролл к
разделителю сделан одноразовым с повтором (`didUnreadScroll`): нода разделителя
может появиться не на первом кадре, если таймлайн ещё догружается.

## 2026-06-15 — Прочтение: «по видимости» вместо «при открытии»

**Что изменилось.** Раньше комната помечалась прочитанной сразу при открытии
(`markRoomRead` до последнего события). Теперь — только по факту видимости:
сообщение считается прочитанным, если реально попало в видимую область и пробыло
там ~1с.

**Зачем.** Открытие комнаты ≠ прочтение. Старое поведение «съедало» непрочитанные
(зашёл — всё обнулилось, даже не увидев). Так делает Telegram.

**Как.** `Timeline` через `IntersectionObserver` (threshold 0.6) + задержка 1000мс
сообщает наверх последнее реально показанное сообщение (`onReadUpTo`), а
`ChatShell` двигает read receipt до него (`markReadUpToEvent`) — только вперёд и
только когда вкладка активна. Плюс разделитель «Новые сообщения» и открытие
комнаты на первом непрочитанном (id первого непрочитанного замораживается на
время открытия через `useFirstUnread`, чтобы линия не уезжала во время чтения).
Файлы: `packages/matrix-core/src/timeline/readReceipts.ts`,
`apps/surf-chat-web/src/features/timeline/{Timeline.tsx,useFirstUnread.ts}`,
`apps/surf-chat-web/src/app/ChatShell.tsx`.

## 2026-06-15 — Кэш Caddy: деплой не подхватывался браузером

**Проблема.** После деплоя фронта браузер продолжал грузить старый JS-бандл —
проверки шли на устаревшем коде, фиксы «не работали».

**Причина.** Caddy отдавал `index.html` без заголовка `Cache-Control`, браузер
кэшировал HTML по эвристике и ссылался на прежний хэшированный бандл.

**Решение.** В `Caddyfile` (на сервере `/opt/synapse-test/`) для сайта
`chat.foxhound.run`:
- `@assets path /assets/*` → `Cache-Control: public, max-age=31536000, immutable`
  (имена файлов содержат хэш — можно кэшировать вечно);
- `@notassets not path /assets/*` → `Cache-Control: no-cache`
  (`index.html` всегда ревалидируется → новый деплой подхватывается сразу).

Деплой Caddy: правка → `caddy validate` → `caddy reload` (graceful, прокси Synapse
не затрагивается). Caddyfile пока живёт только на сервере, в репозитории не отражён.

## 2026-06-15 — Мигание интро в пустом личном чате + бесконечная пагинация

**Проблема.** При входе в пустой DM блок «Имя / Начало личного чата» постоянно
мигал; в фоне шёл шквал запросов пагинации. Замер: ~700k появлений/исчезновений
интро за 5 секунд.

**Причина.** Synapse для DM отдаёт backwards-токен пагинации даже в начале
комнаты. Эффект «заполнить экран историей» считал, что история есть, грузил
страницу без новых сообщений (`paginateBackwards` → `false`) и ставил
`atStart = true`; другой эффект видел, что токен ещё есть (`hasOlder = true`), и
откатывал `atStart = false`. Бесконечный пинг-понг → `RoomIntro` то монтируется,
то нет.

**Решение.** Ввели ref `reachedStart`: как только пагинация вернула «нет новых
сообщений», фиксируем достижение начала, и sync-эффект больше не сбрасывает
`atStart`. Дополнительно `loadOlder` обёрнут в `useCallback` (меньше лишних
перерендеров). Файлы: `apps/surf-chat-web/src/features/timeline/Timeline.tsx`,
`apps/surf-chat-web/src/app/ChatShell.tsx`.

## 2026-06-15 — Дёрганье таймлайна при скролле → убрали виртуализацию

**Проблема.** Лента сообщений дёргалась при любом скролле.

**Причина.** `@tanstack/react-virtual` с оценкой высоты строки 78px при сообщениях
переменной (и большой) высоты: при скролле виртуализатор домерял реальные размеры
и пересчитывал позиции, контент прыгал. Чем больше сообщений в DOM — тем хуже.

**Решение.** Убрали виртуализацию — рендерим загруженные сообщения напрямую в DOM
(`.timeline__item` → `position: relative`). История грузится лениво, поэтому узлов
в DOM немного. Также добавили `overflow-anchor: none` на ленту, чтобы нативный
скролл-анкеринг браузера не конфликтовал с ручным позиционированием при подгрузке
истории. Если в будущем упрёмся в перф на тысячах сообщений — возвращаться к
виртуализации стоит на `react-virtuoso` (он создан под чат-кейс), а не на tanstack.

## 2026-06-15 — Telegram-style вход: заполнять экран историей

**Проблема.** При первом входе в чат было видно только ~10 последних сообщений
(окно начальной синхронизации Matrix), сверху пустота; история подгружалась
только при ручном скролле вверх.

**Решение.** Пока вид прижат к низу и экран не заполнен, лента жадно догружает
историю (`runLoad({ keepBottom: true })`) — чат открывается уже заполненным, как в
Телеграме. Файл: `apps/surf-chat-web/src/features/timeline/Timeline.tsx`.
